Unnecessary BuildConfig Generation

Why This Matters

Library module 'datastore-test' generates BuildConfig.java by default but no source files import it. This creates unnecessary compilation work for every build.

Production Impact

Every module generates and compiles BuildConfig.java even if unused. In 50+ module projects this adds seconds to clean builds and pollutes the classpath.

Resolution 
Disable BuildConfig generation in your android {} block:

buildFeatures {
    buildConfig = false
}
💰 Fewer generated sources, faster compilation, smaller classpath. ⏱️ Save 0.5h/cycle 📄 build.gradle.kts

Circular Dependency Detected

Why This Matters

Module ':core:datastore-test' participates in circular dependency chain(s): :core:datastore-test → :core:datastore-test. Circular deps prevent independent compilation and break incremental builds.

Production Impact

Circular dependencies prevent Gradle from parallelizing module compilation, break Configuration Cache, cause unnecessary recompilations, and indicate tightly-coupled architecture that is impossible to refactor incrementally.

Resolution 
Break the cycle by: 1) Extracting shared code into a new ':core:*' module, 2) Using dependency inversion (interfaces in a shared contract module), 3) Moving the offending dependency behind an API boundary.
💰 Enables parallel compilation, reduces incremental build times by 20-40%, and improves code maintainability. ⏱️ Save 4.5h/cycle 📄 build.gradle.kts
WARNING

Non-Transitive R Class Not Enabled

Why This Matters

android.nonTransitiveRClass=true reduces R.class size and compilation time by not inheriting resources from dependencies into each module's R class.

Production Impact

Each module's R.java contains ALL transitive resources, causing massive regeneration on any resource change in any dependency. Build times scale quadratically with module count.

Resolution 
Add 'android.nonTransitiveRClass=true' to gradle.properties. Fix any R.drawable/R.string references that relied on transitive access.
💰 30-50% faster incremental builds in resource-heavy multi-module projects. ⏱️ Save 0.5h/cycle 📄 gradle.properties
INFO

Dependency Locking Not Configured

Why This Matters

No dependency locking is configured. Without locking, builds may resolve different transitive versions over time, leading to non-reproducible builds.

Production Impact

A transitive dependency could be silently upgraded to a compromised version between builds. Dependency locking ensures byte-for-byte reproducible resolution.

Resolution 
Enable locking: dependencyLocking { lockAllConfigurations() } and commit lockfiles to VCS. Run ./gradlew dependencies --write-locks to generate initial lockfiles.
💰 Reproducible builds, supply chain security, deterministic CI/CD. ⏱️ Save 0.1h/cycle 📄 build.gradle.kts
INFO

JDK Toolchain Not Configured

Why This Matters

No JDK toolchain is configured. Builds may use different JDK versions across developer machines and CI, leading to inconsistent bytecode and potential compatibility issues.

Production Impact

Different JDK versions produce different bytecode optimizations. Without toolchain pinning, a developer on JDK 21 may produce incompatible artifacts for a JDK 17 target.

Resolution 
Add kotlin { jvmToolchain(17) } or java { toolchain { languageVersion.set(JavaLanguageVersion.of(17)) } } to your build file.
💰 Consistent builds across all environments, automatic JDK provisioning. ⏱️ Save 0.1h/cycle 📄 build.gradle.kts

Why This Matters

Module 'datastore-test' has no test files in src/test, src/androidTest, or src/commonTest. This module is completely untested.

Production Impact

Untested modules are invisible risk vectors. Bugs ship silently, regressions go undetected, and refactoring becomes dangerous. In large projects, dark modules become unmaintainable over time.

Resolution 
Add at least unit tests for public API classes. Create src/test/kotlin/ directory and add basic tests using JUnit5 or KotlinTest. Target >60% line coverage for business logic modules.
💰 Early bug detection, safe refactoring, reduced QA cycle time. Each caught regression saves 2-8h of developer time. ⏱️ Save 2.0h/cycle
